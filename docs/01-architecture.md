# 01 · 시스템 아키텍처

## 데이터·제어 흐름

```text
[.xyz 후보들 + 메타(역할/전하/대조군)]
        │
        ▼
┌───────────────────┐
│  xTB Runner       │  GFN2-xTB + geometry opt
│  (Python)         │
└─────────┬─────────┘
          │ raw log
          ▼
┌───────────────────┐
│  Log Parser       │  필요 수치만 추출
│                   │──▶ candidate_results.json  ◀── SSOT
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  MCP Tool Server  │  calculate_* / evaluate_* / report_*
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  Multi-Agents     │  동일 JSON, 역할별 점수·상태
│  (rule-based)     │
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│  Scoring + Report │  가중치 점수, include/caution/exclude
│  (+ optional LLM) │
└─────────┬─────────┘
          │
          ▼
   [실험 설계 보고서] + [홍보 웹 페이지]
```

## 계층 책임

| 계층 | 책임 | 하지 않는 것 |
|------|------|--------------|
| xTB Runner | 계산 실행, 원시 로그 보관 경로 기록 | 추천 판단 |
| Parser | 총에너지·전하·쌍극자·수렴 여부 → JSON | 자연어 해석 |
| MCP Server | 도구 API로 파이프라인 노출 | 임의 수치 생성 |
| Agents | 관점별 점수·요약·상태 | JSON에 없는 수치 발명 |
| Critic | 과잉해석·근거부족·방법론 한계 | 단독으로 순위 조작 |
| Scorer | 가중치 합산·제외 규칙 적용 | LLM 문장으로 점수 덮어쓰기 |
| LLM (optional) | 보고서 문장 다듬기 | 순위·제외 결정 |
| Web | 시스템 소개·결과 전시 | 자체적으로 계산 재해석 |

## 핵심 불변조건

1. 에이전트·보고서·UI가 읽는 계산값은 **파서가 만든 JSON만** 사용한다.
2. LLM 응답이 JSON 수치와 충돌하면 **JSON을 이긴다.**
3. 대조군은 점수표에 나타날 수 있으나, **최종 추천 집합**에서는 실험 목적(예: 양전하 필터)에 맞는 후보만 대상으로 한다.
